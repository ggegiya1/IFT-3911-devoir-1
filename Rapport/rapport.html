Tâche 0. Diagramme de cas d'utilisation concernant le client et la déscription des cas d'utilisation

Premièrement, on à créé une diagrame de cas d'utilisation et la déscription détailée pour chaque cas d'utilisation. Cela nous a aidée de mieux comprendre la structure qu'on devait implementer.



Cas 1: Vérifier des vols disponibles

Acteur : le client
Post-condition : le système affiche la liste des vols avec le détail pour chacun : date et heure de départ, durée du vol, heure d'arrivée, compagnie aérienne, numéro du vol, prix, nombre de sièges disponibles pour la section.

Scénario principal (succès) :
1. Le client commence une nouvelle recherche
2. Le client saisit l’aéroport d'origine, la destination, une date et la classe désirée
3. Le système affiche la liste des vols avec le détail pour chacun : date et heure de départ, durée du vol, heure d'arrivée, compagnie aérienne, numéro du vol, prix, nombre de sièges disponibles pour la section

Scénario alternatif :
3a : aucun siège disponible pour les données saisies
1. Le système retourne une liste vide (soit une message)

Cas 2: Réserver un siège

Acteurs : le client
Pré-conditions :
- il y a au moins un siège disponible dans un vol donné

Post-conditions :
- le système fourni au client le numéro de réservation du siège choisi
- le système marque le siège choisit n’étant plus disponible

Scénario principal (succès):
1. Le client commence une réservation
2. Le client saisit le numéro du vol choisit dans le système
3. Le système affiche les sieges disponibles
4. Le client choisit un siège et saisit le numéro du siège
5. Le système marque le siège choisit RÉSERVÉ
6. Le système présente au client le numéro de réservation du siège choisi

Cas 3: Payer un siège

Acteur : le client
Pré-conditions : le client possède un numéro de réservation
Post-conditions :
- Un siège réservé devient assigné à un passager
- Une confirmation de paiement est produite

Scénario principal (succès):
1. Le client commence un payement
2. Le client saisit le numéro de réservation du siège
3. Le système vérifie si le numéro de réservation est valide :
- la réservation existe dans le système
- la réservation n’est pas expirée (il y a moins de 24 après la réservation)
- le siège réservé n’est pas confirmé
4. Le système affiche le montant à payer et propose au client de payer
5. Le client saisit les données personnelles (non, âge, adresse etc.)
6. Le client effectue un payement par carte de crédit
7. Le système communique au système de payement externe et demande d’autoriser le payement.
8. Le système assigne le siège au client  (change le statut de siège de RÉSERVÉ à CONFIRMÉ)
9. Le système enregistre la vente
10. Le système génère un reçu au client

Cas 4: Annuler une réservation non confirmée

Acteur : le client
Pré-condition : le client possède un numéro de réservation
Post-condition : le siège réservé redevient disponible

Scénario principal (succès):
1. Le client demande une annulation de réservation
2. Le client saisit le numéro de réservation du siège
3. Le système vérifie si le numéro de réservation est valide :
- la réservation existe dans le système
- la réservation n’est pas expirée (il y a moins de 24 après la réservation)
- le siège réservé n’est pas confirmé
4. Le système annule la réservation et change le statut du siège réserve de RÉSERVÉ à DISPONIBLE
5. Le système confirme au client que l’opération a été effectuée

Cas 5: Changer une réservation non confirméee

Acteur : le client
Pré-conditions : le client possède un numéro de réservation
Post-condition :
- le siège réservé redevient disponible
- le systéme propose au client à faire une nouvelle réservation

Scénario principal (succès):
1. Le client demande un changement de réservation
2. Le client saisit le numéro de la réservation actuelle
3. Le système vérifie si le numéro de réservation est valide :
- la réservation existe dans le système
- la réservation n’est pas expirée (il y a moins de 24 après la réservation)
- le siège réservé n’est pas confirmé
4. Le système annule la réservation et change le statut du siège réserve de RÉSERVÉ à LIBRE
5. Le système confirme au client que l’opération a été effectuée et propose au client à effectuer une nouvelle réservation (Cas 1: Vérifier des vols disponibles)

- le siège est confirmé

Contrat pour afficherVols
Opèration : afficherVols(aéroportDépart, aéroportDéstination, date, classe).
Référence : Cas d'utilisation Verifier vols disponibles.
Préconditions : Il n'y a pas de liste de vols.
Postconditions :
  • Une instance listeVols de ListeVols est créée.
  • listeVol est associée avec Vol.

Contrat pour RéserverSiège
Opèration : RéserverSiège (vol, priorité).
Référence : Cas d'utilisation Réserver siège.
Préconditions : Il n'y a pas de réservations et il y a des sièges disponibles pour un vol donné.
Postconditions :
  • Une instance réservation de Réservation est créée.
  • réservation est associée avec Siège d'un Vol.

Contrat pour saisirSiège
Opèration : saisirSiège (numéroSiège)
Référence : Cas d'utilisation Réserver siège.
Préconditions : La valeur de l'attribut noSiègesDispo du Vol est > 0 et le siège à réserver est disponible.
Postconditions :
  • L'attribut étatRéservation du siège devient réservé (ou true).
  • L'attribut noSiègesDispo du Vol décremente sa valeur.

Contrat pour PayerSiège
Opèration : payerSiège (donnéesPersonnelles, modePayement).
Référence : Cas d'utilisation Payer siège.
Préconditions : Le siège n'est pas payé et il est en état réservé.
Postconditions :
  • Une instance payement dans Payement est créée.
  • Une association entre Payement et Réservation est créée.
  • L'instance état du Siège change son état en payé.
  • Les attributs de l'instance du Payement ont enregistrés les données de la transaction.

Contrat pour demanderAnnulation
Opèration : demanderAnnulation (noConfirmation)
Référence : Cas d'utilisation Annuler réservation nonconfirmée.
Préconditions : Une reservation est active, mais elle n'est pas encore payée.
Postconditions :
  • Lattribut état de l'instance du Siège redevient disponnible.
  • La valeur de l'attribut noPlacesDispo du Vol incremente.
  • L'instance réservation de Réservation est éffacée.

Contrat pour demanderChangement
Opèration : demanderChangement (noRéservation)
Référence : Cas d'utilisation Changer réservation nonconfirmée.
Préconditions : Une reservation est active, mais elle n'est pas encore payée.
Postconditions :
  • Lattribut état de l'instance du Siège redevient disponnible.
  • Lattribut état de l'autre instance du Siège devient reservé.

Tâche 6. Contraintes OCL

1) L'aéroport de départ et d'arrivée sont différents.

context DéscriptionVol
   inv: self.départ <> self.arrivée

2) La partie alphabétique de l'ID d'un vol est unique à chaque compagnie
   et la partie numérique est unique à chaque vol au sein de la même compagnie.

context CompagnieAériene
   inv: let
            first2chars = self.vol.id.at(0) + self.vol.id.at(1)
            last_chars  = self.vol.id.excludes(first2chars)
          in
            self.allInstances() -> forAll(c1, c2: CompagnieAériene
               | c1 <> c2 implies c1.first2chars <> c2.first2chars)
            self.vol.allInstances() -> forAll(v1, v2: Vol
               | v1 <> v2 implies v1.last_chars <> v2.last_chars)

3) Tous les sièges d'une même section ont le même prix dans la même compagnie.

context CompagnieAériene
   inv: self.vol.avion.séction.siège.allInstances()
      -> forAll(s1, s2: self.vol.avion.séction.siège
         | s1 <> s1 implies s1.prix = s2.prix)

4) Réserver ne peut se faire que pour un siège libre. Identifiez la postcondition de l'opération réserver().

context Reservation :: réserver (noSiege)
   pre:  self.vol.avion.section.siège.étatRéservation = "disponible"
   post: self.noRéservation <> null
   post: self.tempsRéstantRéservation = 24
